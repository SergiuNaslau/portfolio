Situation
A backbone built on
waiting.

In the realm of 3D project data management, the focus turned to revamping an aging code base that had been the backbone of data-serving capabilities. Clients once faced delays of 2–3 hours — and occasionally days — for update requests.

This dependency on support staff had been tackled head-on. The introduction of a project-based configuration module stands out as the perfect solution: it empowers project managers and analysts to tailor the look and feel of their data, providing a quicker and more responsive environment.

The current system faces significant challenges that hinder both client autonomy and developer efficiency. Clients encounter prolonged waiting times and difficulties configuring project scopes independently — relying on the developer and support staff assistance.

Task · Problem Space
Six friction points
across the system

From user interviews and usage analysis, six compounding problems emerged — affecting clients, developers, and analysts alike.

01
Time zone dependency
Varying time zones across client clientele created unpredictable operational hours and support bottlenecks, making updates feel unreliable.
02
Restricted server access
High costs of operational time on the platform meant developers had restricted access due to financial constraints — already over-stretched.
03
Outdated code base
The presence of an outdated code base contributed to bottlenecks that impede the seamless implementation of evolving solutions.
04
No customisation tools
Lacking customisation features, clients resorted to creating tables in Excel with desired settings and copying them into the system manually.
05
Copy-paste readability issues
Copy-pasting poses challenges including unclear data interpretation and cramped columns that hinder readability across the platform.
06
Developer-only admin
Project management handled by developers for security, not managers — creating overhead and slowing response to client needs.
Action · The Solution
A three-layer approach
for each stakeholder

A comprehensive solution demanded a three-layered approach for optimal functionality — each layer addressing a distinct stakeholder group and their specific needs.

👤
Layer 1 · Client Solution
Project-based configuration module
As a designer, substantial input went into crafting the client solution. The module empowers project managers and clients to customise the appearance of data according to the unique needs of each project — offering 7 settable columns and 2 fixed ones (settings and project title), plus the ability to configure columns to suit their project requirements. No waiting, no Excel workarounds.
⚙️
Layer 2 · Management Solution
Admin rights transferred to project managers
The management solution introduces important admin privileges — transitioning from developer to project managers. Project managers can now gain full control over data access, diminishing the reliance on developers. By boosting access points, the system becomes less dependent on external entities constrained by different time zones. This streamlines access and improves overall admin management.
🔒
Layer 3 · Security Solution
Responsibility shifted, risks reduced
The security solution involves a code update that redistributes responsibility from developers to managers. This strategic move makes potential issues down the line less dramatic and more manageable. A localised learning curve ensures managers can effectively handle and address challenges — contributing to a more resilient and secure system architecture.

Reflections
What this project taught
Design for the right owner
Moving admin rights from developers to project managers wasn't just a convenience — it was the structural change that made everything else possible. Designing for the correct decision-maker unlocks sustainable product ownership.
Excel workarounds are design signals
When users create their own systems in Excel to work around yours, that's not user error — it's a product gap. The configurator directly replaced a workaround that had become standard practice.
Time zones are UX problems
Dependency chains that cross time zones create invisible friction. Reducing reliance on external parties — through self-serve tooling — directly improves perceived responsiveness for the end user.
