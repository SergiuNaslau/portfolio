Situation
Built for one device.
Used by everyone.

The Board feature provides stakeholders with a safe environment to build and present their work — allowing direct access to internal managed files and protecting them from unwanted attention. An essential tool in the life of product stakeholders.

Until 2020, Boards had a dependency on the Microsoft environment. Teams used Microsoft tools to manipulate and bring 3D information to their presentation goals — an approach that was efficient for only one touchpoint: the desktop.

The new stakeholders wanted to iterate in the team, leave feedback, and have a comment system. Because of the different brand partnerships, the need for customisation from a visual perspective made its presence on the goal list. A much-needed update for a mobile-dominated world was overdue.

"Efficient for only one touchpoint — the desktop — in a world that moved on."
2020
Last major update — before mobile became the primary working device for stakeholders
Task · Feature Scope
Three phases.
This project: Phase 1.

Together with the stakeholder team, an implementation plan was developed — taking into consideration both team potential and code limitations — to develop the feature in the right sequence. This project focuses only on Phase 1.

New layout aligned to the grid system
Develop a new layout for the Board system that would align to the new grid system, optimised for mobile and tablet use. Redesign the visual structure from desktop-only to responsive-first.

Action · Challenges & Solutions
The hardest constraint
was the canvas itself

One challenge that persists is the usage and manipulation of 3D resources on mobile devices. It is impossible to edit or manipulate in a cost-effective manner on handheld screens — this remained a known limitation, not a solved one.

As development started to give positive feedback, some stakeholders wanted to add new board components that would push the timeline — causing stress in the production process. The solution was to freeze the scope for Phase 1, with a clear roadmap for future phases. Saying no, with a plan, is still UX work.

Challenge
3D on mobile = impossible to edit
Cost-effective mobile editing of 3D assets remains unsolved at the technology layer — this was accepted and scoped out.
Known limitation — accepted
Challenge
Stakeholder feature creep mid-delivery
New board components requested while Phase 1 was in production. Scope was frozen and deferred to Phase 2 or 3.
Managed — deferred to roadmap

Reflections
What this project taught
Phasing is a design decision
Splitting into 3 phases wasn't a compromise — it was a strategic design choice that protected Phase 1 quality. Knowing what to exclude is as important as knowing what to build.
Layout is the product
Aligning to a grid system isn't a visual preference — it directly drove the 53%→5% error rate improvement. Structure reduces cognitive load before any single feature does.
Dependencies are UX risks
The Microsoft dependency wasn't just a technical constraint — it was a UX bottleneck. Platform independence directly unlocked the mobile experience the product needed.
